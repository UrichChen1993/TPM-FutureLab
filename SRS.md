## 產品範圍與安全需求

### SAFE-01：產品定位

系統僅提供用藥提醒、已確認用藥資訊呈現、紀錄與通知協助，不具備醫療診斷或治療功能。UI、Agent 回覆及 Demo 說明不得讓使用者誤以為系統可取代醫師、藥師或緊急醫療服務。

### SAFE-02：禁止行為

無論產品處於任何發展階段，系統皆不得：

- 診斷疾病或自行判定使用者的健康狀況。
- 自行新增、刪除或變更藥物、劑量、頻率及服藥時間。
- 將口頭回報、藥盒開啟、重量變化或其他感測事件描述為已證明使用者吞服藥物。
- 提供未經醫療專業人員確認的補服、停藥、換藥或加減量建議。
- 僅依據心率、血糖或其他單一生理數據直接判定使用者已用餐。
- 宣稱已聯絡醫療或緊急服務，除非未來確實完成相應串接且取得必要授權。

系統呈現感測結果時，應使用「感測訊號支持已完成」等有限度描述，並保留資料來源、事件時間及可信程度。

### SCOPE-01：MVP 非目標

MVP 不處理自行調整藥物、劑量、頻率或服藥時間。所有提醒只能依據家屬或授權人員已確認且仍在有效期間內的用藥計畫。

## AWS 實作建議

我建議 MVP **實際使用 DynamoDB，Lambda 先列入正式架構，不強制實作**。

理由：

- DynamoDB 可直接證明 Memory 與資料持久化。
- Demo 後重新啟動，仍能查到服藥與事件紀錄。
- Streamlit 可直接展示寫入前後的資料。
- Lambda 在 Live Demo 中較不直觀，增加部署與除錯風險。

MVP 資料流：

```text
IoT 模擬器
    ↓
Streamlit／Python Backend
    ↓
LangChain Agent ──→ Gemini API
    ↓ Tool Calling
DynamoDB
    ├─ 用藥計畫
    ├─ 生理數據
    ├─ 服藥紀錄
    └─ 通知事件
```

正式產品架構則展示：

```text
IoT 裝置
    ↓ MQTT
AWS IoT Core
    ↓
Lambda／EventBridge
    ↓
DynamoDB
    ↓
LangChain Agent Service
    ↓
Gemini API
```

這樣可以清楚區分：

- **PoC 實際做了什麼**
- **正式上線會怎麼擴充**

不要在簡報中讓人誤以為 IoT Core、Lambda、SNS 都已實際部署。

## 功能與技術需求

### UI-01：互動介面

系統應提供 Streamlit 文字對話介面，作為語音 Agent 的互動模擬。

畫面必須標示：

> Text-based simulation of voice interaction

系統不需要執行語音辨識或語音合成。

### AWS-01：資料持久化

系統應實際連接 Amazon DynamoDB，保存：

- 用藥計畫
- 生理數據
- IoT 事件
- 服藥紀錄
- 家屬通知事件
- Agent Session 摘要

若 DynamoDB 無法連線：

- 系統不得假裝寫入成功。
- UI 應顯示資料庫錯誤。
- 對話功能可繼續，但需標示紀錄未持久化。

### AWS-02：真實 DynamoDB 與資料模型

系統必須：
使用 boto3 連接真實 AWS DynamoDB。
讀寫用藥計畫、生理資料、IoT 事件、服藥紀錄及通知。
使用 IAM 最小權限。
從 AWS CLI Profile 或環境變數取得憑證。
禁止將 Access Key 寫入程式碼、.env 範例或 Git。
DynamoDB 無法連線時清楚顯示錯誤，不可假裝寫入成功。
資料表建議
MVP 使用一張表即可：
Table: homewellness-mvp

PK                    SK
USER#user-001         PROFILE
USER#user-001         MEDICATION#med-001
USER#user-001         VITAL#2026-07-17T18:31:00
USER#user-001         IOT_EVENT#2026-07-17T18:30:00#evt-001
USER#user-001         DOSE#2026-07-17#med-001#DINNER
USER#user-001         NOTIFICATION#2026-07-17T19:00:00
USER#user-001         SESSION#session-001
這種設計可用一個使用者 Partition 查詢其相關紀錄，也方便在面試中解釋 DynamoDB 是依「存取模式」設計，而不是照關聯式資料表拆分。
Demo 備援
正式 Demo 預設連接 DynamoDB，但保留：
DATA_BACKEND=dynamodb
DATA_BACKEND=memory
若現場網路或 AWS 驗證失敗，可切換成記憶體模式。畫面必須明確標示目前後端，不能把備援模式說成 AWS 連線成功。
建議實際展示的 DynamoDB 證據
在 Streamlit 側邊顯示：
Backend：AWS DynamoDB
AWS Region
最近一次讀寫時間
最近一次寫入的服藥狀態
Request ID 或事件 ID

### AI-01：模型抽象

系統應透過 LangChain 的模型介面呼叫 Gemini，不得將核心流程綁死在特定模型。

建議保留以下設定：

```text
LLM_PROVIDER
LLM_MODEL
LLM_API_KEY
```

如此可在 Gemini 無法使用時替換其他模型，而不修改業務規則與 Tool 定義。

Gemini 的實際模型名稱與免費額度可能變動，開始實作時再依當時官方資訊確認，不要在 SRS 寫死。

### NOTIFY-01：家屬通知模擬

當規則引擎產生通知時，系統應：

1. 將通知寫入 DynamoDB。
2. 在 Streamlit 家屬通知區顯示。
3. 標示事件時間、原因與嚴重程度。
4. 不實際傳送 SMS、Email 或電話。
5. 不宣稱已聯絡真實家屬或緊急單位。
6. 通知文字須以中文說明目前是第幾次提醒後仍未回應（第1次／第2次），不得只顯示英文狀態碼（如 `missed`）。
7. medium／high 嚴重程度須以不同顏色呈現（medium 為警示黃、high 為錯誤紅），讓家屬一眼分辨緊急程度。

### OCR-01：藥單 OCR 情境模擬

系統應提供固定、可重現的「拍攝／選擇藥單圖片 → OCR 候選 → 藥品外觀參考 → 家屬逐筆校對與確認 → 啟用用藥計畫」展示流程。

- 系統應接受相機拍攝或檔案上傳的圖片作為流程輸入。
- Demo 必須使用固定資料模擬 OCR 辨識結果；上傳圖片不得被描述為已送交真實 OCR 服務處理。
- 畫面必須清楚標示為模擬，不得宣稱已執行真實醫療 OCR。
- 辨識失敗、無候選或資料不完整時，家屬仍可手動新增或修正資料，不得阻斷建檔流程。

### OCR-02：候選資料與編修

每筆模擬辨識候選至少應顯示：

- 藥名。
- 每次劑量。
- 頻率。
- 服用時機。
- 有效期間。
- 辨識信心值。

家屬可新增、修改、補充或刪除候選資料。任何新增或修改後的候選在再次確認前都必須維持 `NEEDS_REVIEW`，不得直接成為提醒來源。

服用時機只允許「飯前」、「飯後」或「固定時間」；選擇「固定時間」時，至少必須提供一個具體時間。

### OCR-03：逐筆確認與啟用驗證

- OCR 候選初始狀態必須為 `NEEDS_REVIEW`。
- 家屬或授權人員必須逐筆確認必要資料，系統不得批次自動確認 OCR 結果。
- 每筆用藥計畫的必填資料為藥名、每次劑量、頻率、服用時機及有效期間；固定時間型計畫另須具體時間。
- 任一必填資料缺漏或格式無效時，系統不得啟用該筆計畫，並須明確列出缺少或有誤的欄位。
- 通過驗證並經授權人員確認後，系統才可將候選寫入 `confirmed=True` 的用藥計畫並啟用提醒。
- IoT 主動提醒只能引用有效期間內且已確認、已啟用的用藥計畫；不得引用待確認、已刪除、未生效或已失效資料。
- 已確認計畫如被修改，變更內容必須重新確認後才可成為提醒依據。

### OCR-04：藥品外觀參考

- 藥品圖片僅能由精確藥名對應可信資料源，並顯示來源及識別碼；無可靠精確結果時不得以相似外觀或生成圖片代替。
- 畫面必須標示外觀圖片僅供核對，不能作為藥品身分或服法的唯一依據。

### AUDIT-01：用藥計畫稽核紀錄

系統應保留每筆用藥計畫的：

- 建立者及建立時間。
- 確認者及確認時間。
- 後續每次新增、修改、刪除、確認與啟用操作。
- 變更前後內容、操作者、操作時間及事件 ID。

稽核紀錄不得因計畫被修改或刪除而覆寫或遺失。

### DEVICE-01：智慧藥盒定位

智慧藥盒應定義為外接選配裝置，透過統一 IoT 事件格式與系統整合。

主系統不應依賴智慧藥盒才能運作：

- 沒有藥盒：允許 `self_reported`
- 有藥盒事件：允許 `sensor_supported`
- 藥盒離線：維持語音提醒，不得標記為硬體確認

### PROACTIVE-01：主動用餐詢問觸發條件

- 晚餐時段定義為 18:00–19:00（左閉右開）。
- 觸發途徑二選一，皆可獨立觸發同一句「王伯伯，您吃完晚餐了嗎？」：
  - 系統時間進入晚餐時段，且當日尚未透過「時間途徑」發送過詢問。
  - 居家感測器（模擬）回報餐桌區域停留事件。
- 時間途徑每日最多觸發一次；IoT 事件途徑依個別事件各自觸發，不做每日次數限制。
- 觸發僅代表「適合詢問」，不得視為已確認用餐，措辭需維持疑問句。

## Demo 最終主線

> 主動用餐詢問的時間途徑（PROACTIVE-01）為背景規則，不在下列主線內獨立列出；可於 Streamlit 側邊欄使用「⏩ 前進 30 分鐘」單獨驗證。

1. 家屬拍攝或上傳藥單圖片；UI 明確標示使用固定資料模擬 OCR。
2. 系統顯示 `NEEDS_REVIEW` 候選、信心值及具有來源標示的藥品外觀參考。
3. 家屬逐筆檢查並修正藥名、每次劑量、頻率、服用時機及有效期間。
4. 系統先展示缺欄或固定時間未填時無法啟用，再於資料完整且人工確認後建立已啟用計畫。
5. Streamlit 模擬晚餐時段與餐桌停留事件。
6. Agent 主動詢問是否已用餐，不以 IoT 或單一生理數據直接判定用餐狀態。
7. 使用者輸入：「吃完了，但今天有點頭暈。」
8. Agent 呼叫 `get_current_vitals()`，Tool 從 DynamoDB 讀取模擬的血壓與心率。
9. 規則引擎進行風險分級，Agent 顯示資料來源及量測時間並進行安全詢問。
10. Agent 在規則允許時，僅依已確認且有效的計畫繼續飯後用藥提醒。
11. 模擬外接藥盒開啟及重量下降，系統寫入 `sensor_supported` 服藥紀錄。
12. Streamlit 以「感測訊號支持已完成」呈現更新結果，不宣稱已證明吞服。
