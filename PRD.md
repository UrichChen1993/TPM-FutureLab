# PRD：主動式健康關懷與用藥提醒

## 1. 產品概述

HomeWellness Companion 結合用藥計畫、居家感測資料及語音 AI Agent，在適當情境主動關懷長者、提醒服藥並記錄結果。若發現持續漏服或健康異常，系統依規則通知家屬。

### 核心價值

> 從固定時間鬧鐘，升級為理解生活情境、能確認結果的主動式照護。

---

## 2. 目標使用者

### 主要使用者：獨居或白天獨處的長者

- 需要每日規律服藥
- 可能忘記服藥或不熟悉手機操作
- 偏好自然、簡短的語音互動

### 次要使用者：家屬照護者

- 協助建立及確認用藥計畫
- 想知道長者是否按時服藥
- 只希望收到需要介入的通知，避免通知疲勞

---

## 3. 使用者問題

### 長者

- 退化導致看不清楚藥袋資訊。
- 剛換藥或領到新藥時，可能不確定每次劑量、頻率與服用時機。
- 固定時間提醒不一定符合實際用餐時間。
- 可能記不得是否已經服藥。
- 身體不舒服時，不知道該提供哪些資訊。

### 家屬

- 無法持續確認長者是否按時服藥。
- 不知道未服藥是單純忘記，還是身體不適。
- 過多通知容易造成焦慮及忽略真正異常。

---

## 4. 產品目標

MVP 要驗證四件事：

1. 家屬可用拍攝或上傳藥單快速建立候選用藥資料，並在人工核對後啟用。
2. 系統能根據時間與 IoT 事件主動發起對話。
3. Agent 能理解長者回應，並呼叫工具取得即時健康資料。
4. 系統能記錄服藥結果，並依風險規則升級通知。

### 產品定位與安全界線

本產品僅提供用藥提醒與相關資訊協助，不具備醫療診斷或治療功能。無論產品處於任何發展階段，皆不會：

- 診斷疾病或判定使用者的健康狀況
- 自行變更藥物、劑量或服藥時間
- 宣稱或證明使用者已實際吞服藥物
- 取代醫師、藥師或緊急醫療服務
- 僅依據心率、血糖等單一生理數據，直接判定使用者是否已用餐

# 非目標

MVP 不處理：

- 自行調整藥物、劑量或服藥時間

---

## 5. 核心 User Stories

### US-01：建立用藥計畫

> 身為家屬，我希望拍攝藥單並確認辨識出的藥物資訊，以便快速建立正確的用藥計畫與提醒。

驗收條件：

- 家屬可拍攝或選擇藥單圖片，啟動藥單辨識流程。
- Demo 使用固定資料模擬 OCR 辨識結果，不宣稱已串接真實 OCR 服務。
- 系統顯示辨識出的藥名、劑量、服用時機及有效期間。
- 家屬可修改、補充或刪除辨識結果。
- 服用時機可設定為「飯前／飯後／固定時間」；選擇固定時間時，必須設定具體時間。
- OCR 辨識結果只能是「待確認」狀態，不得直接啟用提醒。
- 家屬逐筆確認必要資料後，才可啟用用藥計畫。
- 必填資料不完整時，系統不得啟用，並提示缺少的欄位。
- 系統保留建立者、確認者、建立時間，以及後續修改紀錄。
- 辨識失敗或資料不完整時，家屬仍可手動輸入或修正。

Demo 可接收相機拍攝或上傳圖片，但以固定資料模擬 OCR 辨識；顯示每次劑量、頻率、服用時機、信心值及可信來源的藥品外觀參考。辨識結果必須經家屬逐項核對後才可啟用；不宣稱已串接真實 OCR 服務，也不得單靠外觀圖片判定藥品。

### US-02：主動確認用餐狀態

> 身為長者，我希望系統在可能用餐後自然詢問，而不是在不適合的時間直接催促服藥。

驗收條件：

- 系統時間進入 18:00–19:00 晚餐時段、或收到模擬 IoT 事件時，皆會主動觸發詢問（兩者擇一即可，不需同時發生）。
- 時間途徑每日僅觸發一次；IoT 事件途徑依個別事件各自觸發。
- IoT 訊號只觸發對話，不直接判定使用者已用餐。
- 使用者回答「還沒吃」時，可延後提醒。
- 使用者回答不明確時，Agent 應再次簡短確認。

### US-03：提醒並記錄服藥

> 身為長者，我希望在確認用餐後收到清楚的服藥提醒，並讓系統記錄完成狀態。

驗收條件：

- Agent 說明本次應服藥物及已確認的用法。
- 使用者可用語音回報完成。
- 系統可接收模擬的藥盒開啟與重量變化事件。
- 每次紀錄包含時間、來源與可信程度。
- Agent 不提供未經確認的補服或停藥建議。

### US-04：身體不適時讀取健康資料

> 身為長者，當我表示頭暈時，希望 Agent 能取得目前的生理資料並提供安全的下一步協助。

驗收條件：

- Agent 能辨識「頭暈」等不適描述。
- Agent 呼叫工具讀取最新的血壓、心率與量測時間。
- Agent 先詢問是否存在預先定義的危險警訊。
- 風險分級由規則執行，不由 LLM 自由判斷。
- 資料過期或裝置離線時，Agent 必須清楚說明無法取得可靠讀值。

### US-05：未完成時逐級提醒

> 身為家屬，我希望只有在長者持續未完成服藥或伴隨異常狀態時收到通知。

驗收條件：

- 首次未完成時，系統先延後再次提醒。
- 超過設定時間仍未完成，記錄為待確認或漏服。
- 系統依設定通知家屬。
- 單次漏服不直接視為緊急事件。
- 只有同時出現危險訊號或長時間無法聯絡時，才進入緊急處理流程。

## 6. 核心使用者旅程

### 前置設定

1. 家屬建立或確認用藥計畫。
2. 系統保存服藥時間、用餐條件與通知規則。
3. 長者與家屬同意資料使用及通知對象。

### 主動關懷

1. 系統時間進入 18:00–19:00 晚餐時段（此步驟本身即可直接觸發步驟 4 的詢問）。
2. 居家感測器回報餐桌區域停留事件（另一條可獨立觸發相同詢問的路徑）。
3. 系統判定「適合詢問」，而非「確定已用餐」。
4. Agent 主動詢問：「王伯伯，您吃完晚餐了嗎？」

### 對話分支

- 回答「還沒」：詢問何時再次提醒。
- 回答「吃完了」：進入飯後用藥提醒。
- 回答「吃完了，但有點頭暈」：呼叫生理資料工具並進行安全檢查。
- 沒有回應：稍後再提醒；持續無回應才通知家屬。

### 結果確認

1. 長者口頭回報已服藥。
2. 智慧藥盒回報開啟與重量變化。
3. 系統記錄完成狀態及可信程度。
4. 家屬端更新服藥紀錄。

---

## 7. 對話 Vibe 原則

### 語氣要求

- 溫暖，但不過度擬人化
- 簡短，一次只問一件事
- 不責備、不說教
- 重要資訊要重述確認
- 清楚揭露資料無法取得或判斷不確定
- 不使用恐嚇式醫療語言

### 用語範例

建議：

> 「王伯伯，您吃完晚餐了嗎？」

> 「了解，我半小時後再提醒您。」

> 「我讀到的是剛才的量測資料。您現在有胸痛或呼吸困難嗎？」

避免：

> 「系統判斷您已經吃飯，請立刻吃藥。」

> 「您的情況應該沒問題。」

> 「您怎麼又忘記吃藥了？」

## 8. 功能需求

### P0：MVP 必須具備

- 預先建立的長者與用藥資料
- 時間或模擬 IoT 事件觸發主動對話（兩者擇一即可，各自獨立觸發與去重）
- 文字或語音對話介面
- LangChain Agent 工作流
- 對話短期記憶
- 歷史用藥紀錄查詢
- 生理資料 Tool Calling
- 智慧藥盒事件模擬
- 服藥狀態與事件紀錄
- 再次提醒與家屬通知模擬
- 規則式安全分流

### P1：下一階段

- 真實藥袋／處方箋 OCR 服務與影像前處理
- 家屬確認介面
- 真實 IoT 裝置串接
- 語音辨識及語音合成
- 個人化用餐與提醒時段
- 家屬通知管道整合

### P2：未來能力

- 醫院或藥局處方資料整合
- 多照護者權限管理
- 長期健康趨勢分析
- 多裝置與離線模式

## 9. 狀態與可信程度

### 用餐狀態

- `suspected`：時間及 IoT 情境符合
- `confirmed`：使用者明確確認
- `not_eaten`：使用者表示尚未用餐
- `unknown`：沒有回應或回答不明確

### 服藥狀態

- `scheduled`：尚未進入提醒時段
- `due`：應服藥
- `reminded`：已提醒
- `self_reported`：使用者口頭表示已服藥
- `sensor_supported`：語音回報加上藥盒事件
- `missed`：超過設定時間仍未完成
- `needs_review`：資料矛盾或需要家屬確認

建議對外說「感測訊號支持已完成」，不要說「系統證明已服藥」。

---

## 10. 安全與信任原則

- LLM 不自行制定醫療規則。
- 用藥資訊只能來自已確認的用藥計畫。
- Agent 不建議自行加量、補吃、停藥或換藥。
- 生理數據必須附上來源與量測時間。
- 異常分流採可測試、可稽核的規則。
- 通知與資料存取須取得使用者同意。
- 家屬僅能查看獲授權的資料。
- 所有提醒、工具呼叫及通知均保留稽核紀錄。

---

## 11. MVP 成功指標

PoC 階段先驗證流程，不宣稱臨床成效：

- 主動對話成功觸發率
- Tool Calling 成功率
- 服藥流程完成率
- 從 IoT 事件到 Agent 回應的延遲
- 正常、未回應、身體不適三種情境的流程正確率
- 不安全用藥建議攔截率
- 家屬通知是否依規則正確觸發

## 12. Live Demo 驗收情境

Demo 應在 3–5 分鐘內完成：

1. 模擬拍攝藥單與 OCR 辨識，顯示待確認候選。
2. 家屬確認辨識結果，建立已確認的用藥計畫。
3. 模擬晚餐時段與餐桌感測事件。
4. Agent 主動詢問是否用餐，並說明飯後應服藥名與劑量。
5. 使用者回答：「吃完了，但今天有點頭暈。」
6. LangChain 呼叫 `get_current_vitals()`。
7. Agent 顯示資料來源與量測時間，並進行安全詢問。
8. 模擬藥盒開啟及重量下降。
9. 系統記錄 `sensor_supported`，同步更新家屬端狀態。
